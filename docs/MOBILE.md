# iSoyle — план мобильного приложения (Android)

Отдельный план разработки Android-приложения iSoyle. Базируется на решениях из
[DEPLOYMENT.md](DEPLOYMENT.md) (Part B) и уточнён под выбранную стратегию.

## Решения (зафиксировано)
- **Платформы**: только **Android** (iOS не планируется в обозримом будущем).
- **Технология**: **Capacitor** — оборачивает текущий React-фронтенд в нативную
  оболочку, переиспользует весь код UI. (React Native/Kotlin — избыточно для
  тонкого клиента; TWA — рассмотреть как быстрый запасной вариант, см. риски.)
- **Монетизация**: **реклама (Google AdMob)** — приложение бесплатное.
- **Дистрибуция**: **Google Play** (один канал).
- **Архитектура**: **тонкий клиент** к боевому API `https://isoyle.kz`
  (синтез на сервере; on-device — далёкая перспектива, см. M6).
- **Сборка**: Android Studio на **Windows** (Mac не нужен — iOS не делаем).

## Ключевые следствия и ограничения
1. **Приватность**: текст уходит на сервер → **обязательна политика
   конфиденциальности** (это Фаза 8 из LAUNCH.md — становится *блокером релиза*,
   т.к. Google Play её требует). Плюс форма **Data Safety**.
2. **Реклама = сбор данных**: AdMob собирает рекламный идентификатор/данные
   устройства → отразить в Data Safety, добавить метку **«Содержит рекламу»**,
   настроить **согласие пользователя** (Google UMP SDK).
3. **Rate-limit по IP не подходит для мобильных**: пользователи операторов сидят
   за **CGNAT** и делят один публичный IP. Текущий `slowapi` per-IP либо заблокирует
   много добросовестных пользователей, либо один злоумышленник заденет остальных.
   → нужен **лимит по устройству** (см. M4).
4. **CORS**: нативные запросы идут с origin `https://localhost` (Android WebView).
   Чтобы не ослаблять серверный CORS, использовать **CapacitorHttp** (нативный
   HTTP, минует CORS) вместо `fetch`. Альтернатива — добавить origin приложения в
   `CORS_ORIGINS` (менее предпочтительно).

## Стадии

### M1. Подготовка фронтенда к упаковке
- [ ] Конфигурируемый адрес API: собрать бандл приложения с
      `VITE_BACKEND_URL=https://isoyle.kz` (в вебе оставался пустым → относительные
      пути; в приложении нет своего сервера, нужен абсолютный прод-URL).
- [ ] Перевести сетевые вызовы на **CapacitorHttp** (или обёртку, которая в native
      использует CapacitorHttp, в вебе — `fetch`), чтобы обойти CORS и SSE-нюансы.
      ⚠️ Проверить, как работает **SSE** (`/api/tts/stream`) в WebView; при
      проблемах — временно использовать не-стримовый `/api/tts` в приложении.
- [ ] Проверить **адаптивность** на реальном телефоне (сейчас вёрстка близка к
      responsive): плеер, выбор диапазона, длинный текст, клавиатура.
- [ ] **Offline/ошибки**: экран «нет сети», ретраи, понятные сообщения (API уже
      отдаёт ошибки — прокинуть в UI).

### M2. Обёртка в Capacitor
- [ ] Установить Capacitor, `npx cap init` (appId `kz.isoyle.app`, appName `iSoyle`).
- [ ] Добавить Android-платформу: `npx cap add android`.
- [ ] **Иконка и splash** в казахском стиле (переиспользовать солнце + орнамент из
      `favicon.svg`), тёмная тема.
- [ ] **Нативное скачивание mp3**: заменить `<a download>` на Capacitor
      **Filesystem/Share** (сохранение/«Поделиться» файлом).
- [ ] **Аудио**: HTML5 `<audio>` работает в WebView; опц. позже — нативный
      медиа-плеер с фоновым воспроизведением.
- [ ] Проверить сборку и запуск на устройстве/эмуляторе.

### M3. Интеграция рекламы (AdMob)
- [ ] Создать аккаунт **AdMob**, зарегистрировать приложение, получить
      `App ID` и **ad unit** (баннер + interstitial).
- [ ] Плагин `@capacitor-community/admob`: баннер (внизу) + **interstitial**
      (напр., после N-го синтеза — не раздражать на каждом).
- [ ] **Согласие**: Google **UMP SDK** (consent form) — обязателен для показа
      персонализированной рекламы; учесть требования по несовершеннолетним/региону.
- [ ] На старте — **тестовые ad unit ID** (иначе бан аккаунта за клики по своим
      объявлениям).
- [ ] Разместить **`app-ads.txt`** на `https://isoyle.kz/app-ads.txt` (отдаёт
      Caddy как статику) с издательским ID AdMob — верификация и защита выручки.

### M4. Бэкенд под мобильную нагрузку и защиту от абьюза
> Это доработки серверной части (репозиторий backend), нужные из-за CGNAT и
> бесплатной модели с рекламой.
- [ ] **Лимит по устройству**: приложение генерирует стабильный `device_id`
      (хранится локально) и шлёт его в заголовке; сервер ведёт **квоту на
      устройство** (символов/синтезов в сутки) вместо/дополнительно к per-IP.
      Хранилище квот — in-memory на старте, при росте — **Redis** (пересекается с
      A-5 из спецификации).
- [ ] (Опц., сильнее) **Play Integrity API** — подтверждение, что запрос идёт от
      настоящего приложения, а не от скрипта. Защищает бесплатный CPU-эндпоинт.
- [ ] Оставить общий per-IP лимит как грубый предохранитель, но **основной** —
      per-device.
- [ ] Пересмотреть `TTS_RATE_LIMIT`/квоты под ожидаемый поток из приложения.

### M5. Публикация в Google Play
- [ ] **Play Console** аккаунт ($25 разово), **Play App Signing**, `applicationId`
      `kz.isoyle.app`, схема версий (`versionCode`/`versionName`).
- [ ] **Политика конфиденциальности** (Фаза 8) — обязательна; раскрыть отправку
      текста на сервер и сбор данных рекламой.
- [ ] Форма **Data Safety**: текст → сервер (обработка), AdMob → рекламный ID/данные
      устройства; метка **«Содержит рекламу»**.
- [ ] **Content rating** (анкета), **target API level** (актуальный по требованиям
      Play), листинг: иконка 512, feature graphic, скриншоты, описания **KZ/RU/EN**.
- [ ] Сборка релиза: `npm run build` → `npx cap sync android` → Android Studio →
      подписанный **AAB**.
- [ ] Треки Play: **internal testing → closed testing → production**.

### M6. On-device в будущем (post-MVP, опционально)
Если приватность станет жёстким требованием — конвертировать акустическую модель и
вокодер в **ONNX/TFLite**, зашить один голос (~120 МБ), инференс на устройстве.
Отдельный крупный R&D (совпадает с post-MVP пунктом ONNX из спецификации).

## Порядок и зависимости
1. **M1 → M2** (рабочее приложение поверх боевого API).
2. **M4** (защита бэкенда) — до широкого теста, иначе риск абьюза CPU.
3. **M3** (реклама) — можно параллельно с M2, но с тестовыми ID.
4. **Фаза 8 (privacy policy)** — *обязательный блокер* перед M5.
5. **M5** (публикация) — в самом конце.

## Риски и заметки
- **SSE в WebView**: если стриминг прогресса не заведётся в упаковке — деградировать
  до `/api/tts` (без пошагового прогресса) только в приложении; веб не трогать.
- **Единая функция разбиения предложений** остаётся на бэкенде — приложение, как и
  веб, использует `/api/split` (правило из CLAUDE.md).
- **Нагрузка на один CPU-сервер**: реклама = много бесплатных пользователей →
  синтез CPU-bound. Следить за очередью/квотами; при росте — A-5 (Redis + очередь,
  горизонтальное масштабирование).
- **Быстрый запасной вариант — TWA (Bubblewrap)**: если нужно опубликоваться очень
  быстро, PWA+TWA даёт Android-приложение почти без кода, но без нативной рекламы/
  файлов так удобно, как Capacitor. Основной путь — Capacitor.
- **Не юрист**: политику конфиденциальности/Data Safety перед публикацией стоит
  выверить (особенно из-за рекламы и отправки текста).

## Definition of Done (MVP приложения)
Android-приложение iSoyle в Google Play (production): синтез через боевой API,
нативное скачивание/шеринг mp3, баннер+interstitial AdMob с согласием, квоты по
устройству на бэкенде, политика конфиденциальности и Data Safety оформлены,
локализация листинга KZ/RU/EN.
